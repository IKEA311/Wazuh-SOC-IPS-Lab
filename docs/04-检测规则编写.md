# 04 - 检测规则编写


本文档记录了 15+ 条自定义 Wazuh 检测规则的设计思路、编写过程和调试经验。

---

## 一、规则编写基础

### 1.1 规则文件结构
Wazuh 规则使用 XML 格式，核心元素:
```xml
<group name="custom-attacks,">
  <rule id="100001" level="12">
    <decoded_as>json</decoded_as>
    <field name="event_id">^1$</field>
    <description>可疑进程创建告警</description>
    <group>process_creation,sysmon,</group>
    <options>no_full_log</options>
  </rule>
</group>
```

### 1.2 规则优先级与 ID 分配
- 使用自定义 ID 范围（100000+），避免与官方规则冲突
- Level 0-15 分级策略:

| Level | 含义 | 适用场景 |
|-------|------|---------|
| 0-3   | 信息 | 普通日志记录 |
| 4-6   | 低危 | 扫描检测、信息收集 |
| 7-9   | 中危 | SQLi 尝试、WebShell 连接 |
| 10-12 | 高危 | 漏洞利用、权限提升 |
| 13-15 | 紧急 | 横向移动成功、系统沦陷 |

## 二、自定义规则详解

### 2.1 SQL 注入检测规则

**文件**: `rules/custom-sqli.xml`

```xml
<group name="sql-injection,">
  <rule id="100010" level="8">
    <if_sid>80700</if_sid>  <!-- 基于 Apache 访问日志规则 -->
    <match>select|union|drop|insert|delete|--|%27|%23</match>
    <description>SQL 注入攻击模式检测 - URL 参数包含恶意关键字</description>
    <group>attack,sql_injection,</group>
  </rule>

  <rule id="100011" level="10">
    <if_sid>100010</if_sid>
    <match>sqlmap</match>
    <description>SQLMap 自动化注入工具检测</description>
    <group>attack,sql_injection,sqlmap,</group>
  </rule>

  <rule id="100012" level="12">
    <if_sid>80700</if_sid>
    <regex type="pcre2">\b(\d+)\b.*union.*select.*from|select.*from.*where.*=.*'</regex>
    <description>SQL 注入成功 - 检测到 UNION SELECT 模式</description>
    <group>attack,sql_injection,critical,</group>
  </rule>
</group>
```

**检测效果**: SQLMap 发起请求后立即触发 `100010` 告警，检测到 `sqlmap` User-Agent 时升级为 `100011`。

### 2.3 WebShell 检测规则

**文件**: `rules/custom-webshell.xml`

```xml
<group name="webshell,">
  <!-- FIM 检测 WebShell 文件创建 -->
  <rule id="100030" level="7">
    <if_sid>554</if_sid>
    <field name="file">\.php$</field>
    <description>可疑 PHP 文件创建 - 可能的 WebShell 植入</description>
    <group>fim,webshell,suspicious_file,</group>
  </rule>

  <!-- WebShell 通信流量检测 -->
  <rule id="100031" level="9">
    <if_sid>80700</if_sid>
    <match>eval|base64_decode|system\(|shell_exec|passthru</match>
    <description>WebShell 通信检测 - 请求包含典型 WebShell 函数</description>
    <group>attack,webshell,</group>
  </rule>

  <!-- 低频率长连接 WebShell 检测 -->
  <rule id="100032" level="12">
    <if_sid>100031</if_sid>
    <regex type="pcre2">POST.*shell\.php|POST.*cmd\.php|POST.*eval\.php</regex>
    <description>WebShell 连接确认 - 检测到 POST 请求到疑似 Webshell 页面</description>
    <group>attack,webshell,critical,</group>
  </rule>
</group>
```

### 2.4 MS17-010 漏洞利用检测

**文件**: `rules/custom-ms17-010.xml`

```xml
<group name="ms17-010,">
  <!-- 检测可疑的 SMB 流量 -->
  <rule id="100040" level="7">
    <if_sid>60103</if_sid>
    <field name="event_id">^3$</field>
    <field name="destination_port">445</field>
    <description>SMB 网络连接 - 可能的内网横向移动</description>

  <rule id="100041" level="12">

    <field name="parent_process">.*spoolsv\.exe</field>
    <description>MS17-010 利用检测 - spoolsv.exe 创建异常子进程</description>
    <group>attack,ms17-010,exploit,</group>
  </rule>

  <!-- 检测 meterpreter 常见行为 -->
  <rule id="100042" level="10">
    <if_sid>61647</if_sid>
    <field name="process_name">cmd\.exe</field>
    <field name="parent_process">spoolsv\.exe|lsass\.exe|svchost\.exe</field>
    <description>异常进程链 - 非交互式进程启动 cmd.exe</description>
    <group>attack,anomaly,process_chain,</group>
  </rule>

  <!-- 检测 MS17-010 后门用户创建 -->
  <rule id="100043" level="10">
    <if_sid>60103</if_sid>
    <field name="event_id">^4720$|^4722$|^4732$</field>
    <description>Windows 用户/组变更告警 - 可能的权限维持</description>

</group>
```

## 三、规则调试方法

### 3.1 🐞 踩坑记录: 规则语法错误导致 Manager 无法启动

**问题现象**: 部署新规则后重启 Manager 失败。

**排查过程**:
```bash
# 查看 Manager 日志
docker logs wazuh-docker_single-node-wazuh-manager_1 --tail 50

# 发现错误:
# "Rule ID 100010: Duplicate rule ID"
```

**解决方案**: 
检查发现两个不同的规则文件中使用了相同的 rule ID。重新分配 ID 范围（按主题分段）:
- 100000-100009: 保留
- 100010-100019: SQLi 检测
- 100020-100029: 暴力破解检测
- 100030-100039: WebShell 检测
- 100040-100049: MS17-010 检测
- 100050-100059: 异常进程检测

### 3.2 规则调试命令

```bash
# 测试单个日志事件匹配规则
/var/ossec/bin/wazuh-logtest

# 在 logtest 交互界面粘贴日志行进行测试
# 例如，粘贴以下 Apache 日志进行测试:
# 192.168.232.129 - - [07/May/2026:10:15:30 +0800] "GET /vulnerabilities/sqli/?id=1'+union+select+1,database()--+&Submit=Submit HTTP/1.1" 200 7456 "-" "sqlmap/1.7.2"
```

## 四、规则调优经验

### 4.1 误报处理

实际问题: SQLi 规则 `100010` 误报了正常搜索功能。
```xml
<!-- 增加白名单排除 -->
<rule id="100010" level="8">
  <if_sid>80700</if_sid>
  <match>select|union|drop|insert|delete</match>
  <!-- 排除正常搜索请求 -->
  <regex type="pcre2">(?i)^(?!.*search|.*catalog|.*product).*$</regex>
  <description>SQL 注入攻击模式检测 - URL 参数包含恶意关键字</description>
</rule>
```

### 4.2 频率阈值调优

实际经验值:
- SSH 暴力破解: `frequency="5" timeframe="10"`（10秒5次，适合内网）
- RDP 暴力破解: `frequency="5" timeframe="60"`（60秒5次，RDP 较慢）
- WebShell 通信: `frequency="3" timeframe="300"`（5分钟3次，避免误报）
